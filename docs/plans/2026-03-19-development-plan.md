# Claw-Pet 开发计划

日期：2026-03-19

## 1. 当前状态

项目已经恢复出最小可运行的 Tauri 2 桌面闭环。

已确认：
- `npm run build` 成功
- `npm run tauri:dev` 成功
- `npm run tauri:build` 成功
- `src-tauri` 最小工程已恢复
- React 前端已重新接回桌面壳层

当前主入口：
- 浏览器开发：`npm run dev`
- 桌面开发：`npm run tauri:dev`
- 桌面构建：`npm run tauri:build`

这意味着项目已经跨过“桌面工程不存在”的阶段。后续工作的重点不再是恢复 Tauri 骨架，而是恢复桌宠能力本身。

## 2. 当前问题边界

当前阶段先不处理以下内容：
- 系统托盘
- 开机启动
- websocket 或 dialog 等 Tauri 插件扩展
- 高级窗口效果和系统集成

这样做的原因很简单：这些功能会同时引入平台差异、权限配置和新变量。现在最重要的是先把 Live2D 与 Gateway 两条主业务链路分开恢复，避免问题互相污染。

## 3. 下一阶段目标

下一阶段按以下顺序推进：

### 阶段 A：稳定桌面壳
目标：保持当前 `tauri:dev` 和 `tauri:build` 持续可用。

重点：
- 不破坏现有 `src-tauri` 最小工程
- 保持 Vite 端口 `1420` 与 Tauri 配置一致
- 保持文档与真实运行方式一致

### 阶段 B：恢复 Live2D 桌面渲染
目标：让桌面端稳定显示和交互 Live2D 模型。

状态：2026-03-29 已完成 Stage B 验收。

已确认：
- Live2D 模型在桌面端可见
- 点击宠物可触发动作
- 表情按钮可切换表情
- 加载失败时会显示 fallback preview，而不是空白

重点：
- 检查 `PetDisplay`、`Live2DPetDisplay`、`HaruLoader` 的桌面端表现
- 验证模型路径、资源加载、动作和表情切换
- 确保 Live2D 问题不会阻塞桌面壳启动

### 阶段 C：恢复 OpenClaw Gateway 通信
目标：让桌面端重新具备消息收发与连接状态管理能力。

重点：
- 校验 `src/lib/openclaw/client.ts`
- 校验 `src/state/store.ts`
- 验证连接、断开、重连、消息展示是否符合预期

### 阶段 D：恢复增强能力
目标：在主链路稳定后，再考虑更完整的桌宠体验。

重点：
- Tauri 插件恢复
- 托盘与开机启动
- 高级桌面集成能力

## 4. 执行原则

后续开发遵守以下原则：

1. 先保桌面壳稳定，再恢复业务功能。
2. Live2D 和 Gateway 分阶段恢复，不同时大改。
3. 每完成一个阶段，都用命令回归验证。
4. 文档必须跟随代码更新，避免再次出现“文档领先代码现实”。

## 5. 每阶段验收标准

### 阶段 A 验收
- `npm run tauri:dev` 成功
- `npm run tauri:build` 成功

### 阶段 B 验收
- Live2D 模型在桌面端可见
- 点击或交互动作可触发
- 模型资源路径无错误

### 阶段 C 验收
- Gateway 可连接
- 消息可显示
- 重连逻辑可用

### 阶段 D 验收
- 增强功能不破坏前 3 个阶段结果

## 6. 推荐的近期任务

建议近期优先做：

1. 复查 Live2D 桌面端现状
2. 明确 Gateway 当前可用程度
3. 为下一轮改动补一份更细的实现计划

## 7. 文档用途

这份文档用于记录当前开发主线，帮助后续在恢复 Live2D、Gateway 和桌面增强能力时保持一致方向。若后续目标变化，应直接更新本文件，而不是再新增口径冲突的状态说明。
